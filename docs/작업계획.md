## 1. 아키텍처 결정 요약 (기획서 → 구현 확정)

기획서의 하이브리드 설계를 구현 관점으로 확정한다.

- **플레이어 / 카메라 / UI / VFX** = GameObject·MonoBehaviour 월드
- **잡몹 수천 (스폰·이동·타겟팅·근접판정·사망·렌더)** = ECS 월드
- **전투는 양방향**: 플레이어가 적을 벨 뿐 아니라 **적(몬스터)도 플레이어를 공격**해 플레이어가 피격·사망(패배)할 수 있다. 단, "양방향"은 게임플레이 의미이고, 브릿지 통신은 아래처럼 **각 스트림이 단방향 데이터 흐름**을 유지한다.
- **브릿지는 기획서 "방식 A"로 통일** (스트림별 단방향 데이터 흐름):
  - 플레이어 → ECS: 공격 시 `EntityCommandBuffer`로 **AttackRequest 엔티티** 스폰
  - ECS → 플레이어 ①: **사망 이벤트** → NativeQueue 수집 → 프레임당 1회 메인스레드 소비 (VFX·킬카운트)
  - ECS → 플레이어 ②: **플레이어 피격 데미지 이벤트** → NativeQueue 수집 → 프레임당 1회 합산 소비 (플레이어 HP↓·피격 연출) ← *몬스터의 플레이어 공격*
- **①순진한(naive) 구현은 처음부터 별도 브랜치**로 격리 → 벤치마크 출발점 (§6)

> 이 결정들은 되돌리지 않는다. 브릿지 방식 B(네이티브 큐 push), 플레이어 ECS화 등은 스코프 밖(기획서 §3 제외 목록 = 신성불가침).

## 3. ECS 데이터 모델 설계 (컴포넌트 확정)

시스템을 짜기 전에 데이터부터 확정한다(DOP의 핵심). 아래를 `Components/`에 배치.

```csharp
// 태그
struct Enemy : IComponentData {}                    // 잡몹 식별
struct DeadTag : IComponentData, IEnableableComponent {}  // 사망 마킹(Enableable로 청크 유지)

// 상태 데이터
struct Health   : IComponentData { public float Value; public float Max; }
struct MoveStats: IComponentData { public float Speed; public float StopDistance; }
struct Velocity : IComponentData { public float3 Value; }
struct TargetPos: IComponentData { public float3 Value; }   // 추적 목표(플레이어 위치 스냅샷)

// 적의 대(對)플레이어 공격 스펙  ← 몬스터가 플레이어를 공격
struct EnemyAttack : IComponentData {
    public float Range;     // 이 거리 안이면 공격
    public float Damage;    // 1회 타격 데미지
    public float Cooldown;  // 공격 간격(초)
}
struct AttackCooldown : IComponentData { public float Remaining; }  // 남은 쿨타임(0이하=공격 가능)

// 데미지 버퍼(한 프레임에 여러 히트 누적 가능) — 적이 받는 데미지
struct DamageEvent : IBufferElementData { public float Amount; public float3 Knockback; }

// 브릿지: 플레이어 공격 요청 엔티티 (플레이어 → ECS)
struct AttackRequest : IComponentData {
    public float3 Center; public float Radius; public float Damage; public float3 Dir;
}

// 싱글톤: 플레이어 상태 GO→ECS 프레임 스냅샷 (HP도 포함 → 죽었으면 적이 공격 중단)
struct PlayerState : IComponentData {
    public float3 Position; public float3 Forward; public float Health; public bool Alive;
}

// 싱글톤: ECS→GO 사망 이벤트 수집 핸들 (NativeQueue 래핑)
struct DeathEventQueue : IComponentData { /* NativeQueue<DeathEvent> 핸들 */ }

// 싱글톤: ECS→GO 플레이어 피격 데미지 수집 핸들  ← 몬스터의 플레이어 공격 결과
struct PlayerDamageQueue : IComponentData { /* NativeQueue<PlayerHit> 핸들 */ }
struct PlayerHit { public float Amount; public float3 SourcePos; }  // 큐 요소(피격 연출용 방향 포함)
```

- **Spatial Hash**는 컴포넌트가 아니라 시스템이 소유하는 `NativeParallelMultiHashMap<int, Entity>`(또는 셀→인덱스)로 관리(§4.4).
- `LocalTransform`(Unity.Transforms 기본 제공)을 위치의 단일 소스로 사용. `Velocity`는 이동 시스템이 여기에 적분.

## 4. 시스템 설계 (SystemGroup 순서)

`SimulationSystemGroup` 안의 실행 순서를 명시적으로 고정한다.

```
SpawnSystem                 // 카운트 슬라이더 → 부족분 ECB 스폰
  → PlayerSnapshotSystem    // GO 플레이어 위치·HP·생존을 PlayerState 싱글톤에 기록 (브릿지 IN)
  → SpatialHashBuildSystem  // 이번 프레임 그리드 재구축
  → TargetingSystem         // TargetPos = PlayerState.Position (초기엔 단순 추적)
  → MovementSystem          // Velocity/이동 적분 (+ 이웃 회피는 심화)
  → EnemyAttackSystem       // ★ 적이 플레이어 사거리 내 & 쿨 0이면 PlayerHit 큐에 push, 쿨 tick
  → AttackResolveSystem     // AttackRequest × SpatialHash → DamageEvent 버퍼 기록
  → DamageApplySystem       // DamageEvent 소비 → Health 감소 → 0이하면 DeadTag on
  → DeathSystem             // DeadTag → DeathEventQueue push + ECB 파괴
```

메인스레드 소비(브릿지 OUT)는 `SimulationSystemGroup` 이후(또는 `LateSimulation`)에서 1회 — **DeathEventQueue + PlayerDamageQueue 둘 다** 이때 비운다.

### 4.1 SpawnSystem
- 목표 카운트(UI 슬라이더 → 싱글톤)와 현재 살아있는 수 비교, 부족분을 프리팹 엔티티에서 ECB로 인스턴스화.
- 스폰 위치: 링/원형 분포. `IJobEntity`로 초기화.

### 4.2 MovementSystem (`IJobEntity` + Burst)
- `TargetPos`로 향하는 방향 정규화 × `Speed` → `LocalTransform.Position` 갱신.
- StopDistance 이내면 정지(플레이어 주변 밀집). 이웃 회피(separation)는 Week 2 후반 or 심화.

### 4.3 TargetingSystem
- MVP: 전원 플레이어 1인 추적 → `TargetPos = PlayerState.Position` 대입만. (병종 다양화는 스코프 밖.)

### 4.3b EnemyAttackSystem (★ 몬스터의 플레이어 공격, `IJobEntity` + Burst)
- 대상이 플레이어 1인뿐이므로 **SpatialHash 불필요** — 각 적이 `PlayerState.Position`까지 거리만 계산(O(n)).
- 로직(적 1기당):
  1. `PlayerState.Alive == false`면 아무것도 안 함(사망한 플레이어 공격 방지).
  2. `AttackCooldown.Remaining > 0`이면 `-= dt` 후 종료.
  3. 거리 ≤ `EnemyAttack.Range` 이고 쿨 0이면 → `PlayerDamageQueue`에 `PlayerHit{ Amount=Damage, SourcePos=적 위치 }` **병렬 push**(`.AsParallelWriter()`) → `Remaining = Cooldown` 리셋.
- **밸런싱 주의**: 수천 마리가 동시 접촉하면 데미지가 폭주한다. 대응 옵션(택1, W4에서 튜닝):
  - 쿨다운을 넉넉히(예: 1.0~1.5s) + 데미지 소량 → 자연 상한
  - 또는 메인스레드 소비 시 **프레임당 플레이어 피격 총량에 캡**(예: `min(합계, maxDpsPerFrame)`)
- `DeadTag` 활성(죽는 중) 적은 쿼리에서 제외(`WithDisabled`/`WithNone` 활용).

### 4.4 SpatialHashBuildSystem + 근접 탐색 (★ 벤치마크 핵심)
- 자료구조: `NativeParallelMultiHashMap<int, Entity>` (key = `hash(cellCoord)`).
- 셀 크기 = 공격/회피 반경 기준(대개 1~2m). `Build` 잡: 각 적의 셀 좌표 계산 후 병렬 insert.
- 질의: 대상 위치의 **3×3(2D) / 3×3×3(3D) 셀**만 순회 → O(n²) 회피.
- **④단계 벤치마크의 주인공.** naive 브랜치에서는 이 시스템이 없고 전수 O(n²) 순회로 대체(§6).

### 4.5 AttackResolveSystem (브릿지 판정)
- `AttackRequest` 엔티티들을 순회 → 각 요청의 `Center/Radius`로 SpatialHash 질의 → 반경 내 적의 `DamageEvent` 버퍼에 append.
- 처리 후 AttackRequest 엔티티는 ECB로 파괴(1프레임 수명).

### 4.6 DamageApplySystem / DeathSystem
- DamageApply: 버퍼 합산 → `Health.Value` 감소 → 0이하 시 `DeadTag` 활성.
- Death: `DeadTag` 엔티티를 `DeathEventQueue`에 위치/타입 push → ECB로 파괴.

## 5. 브릿지 상세 (GO ↔ ECS, 방식 A 단방향)

```
[플레이어 MonoBehaviour]                          [ECS World]
 공격 애니 히트 프레임
   └─ EntityManager/ECB로 AttackRequest 스폰 ─────▶ AttackResolveSystem
 매 프레임 위치·HP·생존 기록 ────────────────────▶ PlayerState 싱글톤 (PlayerSnapshotSystem)

 VFX/카메라셰이크/킬카운트     ◀── 프레임당 1회 소비 ── DeathEventQueue    (적 사망)
 플레이어 HP↓·피격연출·패배판정 ◀── 프레임당 1회 합산 ── PlayerDamageQueue (적→플레이어 공격)
```

구현 포인트:
- 플레이어 측에서 ECS 접근은 `World.DefaultGameObjectInjectionWorld.EntityManager` 통해. 스폰은 매 히트마다 소량이므로 EntityManager 직접 or 프레임 ECB.
- `DeathEventQueue` / `PlayerDamageQueue`는 시스템이 소유·할당하고, 플레이어 측 브릿지 MonoBehaviour가 `LateUpdate`에서 **각각 TryDequeue 루프 1회**로 비운다(메인스레드 동기화 지점 최소화).
- **플레이어 피격 처리**(PlayerDamageQueue 소비): 프레임당 dequeue한 `PlayerHit`들을 **합산 → (필요시 캡 적용) → 플레이어 HP 차감** → HP≤0이면 `Alive=false` 세팅하고 패배 연출. 대표 `SourcePos`로 피격 방향 넉백/카메라 셰이크.
- **역참조 없음 유지**: ECS 시스템은 GameObject를 직접 만지지 않는다. 적→플레이어 공격도 결과를 큐 데이터로만 노출하고, 실제 HP 반영은 GO 측이 한다. (플레이어 HP의 소유자는 GO, ECS는 스냅샷만 읽음.)

## 6. 벤치마크 하니스 & 브랜치 전략 (★ 포폴의 심장)

기획서 §5의 5단계를 **측정 가능한 코드 구조**로 만든다.

### 6.1 브랜치 전략
| 브랜치 | 내용 | 용도 |
|---|---|---|
| `main` | 최신 통합(③④⑤ 최적화 반영) | 데모/제출본 |
| `bench/01-naive` | MonoBehaviour + O(n²) 근접탐색 | ①단계 기준선(별도 유지) |
| `bench/02-ecs-single` | ECS 전환, 싱글스레드(Burst off) | ②단계 |
| `bench/03-burst-job` | 병렬 잡 + Burst | ③단계 |
| `bench/04-spatial` | Spatial Hash 도입 | ④단계 |

> ①naive 브랜치는 Week 2에 **처음부터 별도로 뜬다**(기획서 §9 액션 3). 나중에 재현하려 하지 말 것.

### 6.2 측정 하니스 (`Benchmark/`)
- **FrameTimeLogger**: N프레임 warmup 후 M프레임 평균/95퍼센타일 프레임타임(ms) 기록 → CSV.
- **자동 스윕**: 적 수 `[500, 1000, 2000, 3000, 5000, 8000, 10000]`를 순차 세팅하며 각 구간 CSV 한 줄 출력.
- 고정 조건: 같은 씬·같은 카메라·같은 해상도·VSync off·고정 시드.

### 6.3 스프레드시트 컬럼 (`벤치마크/` Obsidian 폴더에 축적)
```
단계 | 적수 | avg_ms | p95_ms | main_thread_ms | worker_ms | draw_calls | gpu_ms | 비고
```
→ "단계별 프레임타임 vs 적 수" 꺾은선 그래프 + Profiler 타임라인 캡처 + Burst Inspector SIMD 캡처가 최종 증거물(기획서 §5).
